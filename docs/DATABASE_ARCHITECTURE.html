Data Flow Patterns

Read Path (cache-first)

1

Client sends GET request

SwiftUI or Streamlit calls GET /api/requests, /api/accounts, etc.

2

API reads from SQLite cache

All GET endpoints read exclusively from local cache.db. Zero Snowflake round-trips for reads.

3

Return JSON to client

Sub-millisecond local SQLite response returned to the calling app.

Write Path (cache-first + background sync)

1

Client sends POST/PUT/DELETE

Create, update, submit, approve, reject, withdraw, revise, delete operations.

2

Write to SQLite immediately

API validates against cache, writes changes to cache.db, returns HTTP 200 to client instantly.

3

Background sync to Snowflake

sync_to_snowflake() runs in a background thread. Writes to TEMP.INVESTMENT_GOVERNANCE using SNOWHOUSE warehouse.

4

Re-hydrate cache from Snowflake

After sync completes, cache is refreshed from Snowflake to pick up server-side changes (e.g., autoincrement IDs).

Why SQLite?

The local SQLite cache exists to solve three problems:

⚡

Instant Response Times

All read operations (request lists, account lookups, user info) serve from local disk instead of round-tripping to Snowflake. Sub-millisecond latency vs. 200-800ms per Snowflake query.

🔄

Optimistic Writes

Users see their changes immediately. The UI updates from the cache while Snowflake sync happens in the background. No spinners or "saving..." wait states.

💰

Reduced Warehouse Cost

Reads never consume Snowflake compute credits. Only writes use the SNOWHOUSE warehouse, keeping costs proportional to actual data changes.

🌐

Shared Backend

Both the macOS SwiftUI app and the Streamlit web app hit the same Flask API server and the same cache. Changes made in either client are visible in both.
