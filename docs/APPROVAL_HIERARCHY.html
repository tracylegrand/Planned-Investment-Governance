Approval Levels

The approval chain consists of five levels. Level 1 is the creator/submitter. Levels 2–5 are sequential approvers, each representing a management tier. The chain culminates with the GVP as final approver.

How Approval Routing Works

1

Initial Routing on Submission

When submitted, the system looks up the creator's manager from cached_users and sets NEXT_APPROVER_NAME to that manager. Sets CURRENT_APPROVAL_LEVEL = 1, STATUS = SUBMITTED.

-- api_server.py: submit endpoint
cur2.execute("SELECT manager_name FROM cached_users WHERE snowflake_username = ?", (created_by,))
next_approver = mrow["manager_name"] if mrow else None
2

Status Transitions Map

Each approval is driven by the status_transitions dictionary which maps the current status to the next status, the approval fields to populate, and the new level.

status_transitions = {
  'SUBMITTED':    ('DM_APPROVED',    'dm_approved_by',  'dm_approved_by_title',  'dm_approved_at',  'dm_comments',  2),
  'DM_APPROVED':  ('RD_APPROVED',    'rd_approved_by',  'rd_approved_by_title',  'rd_approved_at',  'rd_comments',  3),
  'RD_APPROVED':  ('AVP_APPROVED',   'avp_approved_by', 'avp_approved_by_title', 'avp_approved_at', 'avp_comments', 4),
  'AVP_APPROVED': ('FINAL_APPROVED', 'gvp_approved_by', 'gvp_approved_by_title', 'gvp_approved_at', 'gvp_comments', 5)
}
3

Per-Approval Processing

For each approval action:

  1. Read current status to determine the transition
  2. Record the approver's name and title (from VW_CURRENT_USER_INFO via cached session)
  3. Store the approval timestamp and optional comments
  4. Advance CURRENT_APPROVAL_LEVEL to the next level
  5. Transition STATUS to the next state
✓

Final Approval by Jon Beaulier

When Jon Beaulier (APPROVAL_LEVEL=5, IS_FINAL_APPROVER=true, ROLE=GVP) approves a request in AVP_APPROVED status:

STATUS = FINAL_APPROVED GVP_APPROVED_BY = "Jon Beaulier" LEVEL = 5

Approver Actions at Each Level

Every approver (Levels 2–5) has four possible actions on a request that has reached their level.

✅ Approve

Advances the request one level in the chain. Records approver's name, title, timestamp, and comments in level-specific fields.

Result: Next status in chain (or FINAL_APPROVED at Level 5)

↰ Send Back

Returns request to DRAFT for revision. Comments stored in GVP_COMMENTS. Previous approval fields are preserved (not cleared).

Result: DRAFT

✖ Reject

Sets status to REJECTED. Creator can revise the business case and resubmit or save as draft.

Result: REJECTED (revisable)

⛔ Deny

Sets status to DENIED with comments in GVP_COMMENTS. No further action path. Terminal state.

Result: DENIED (terminal)

Pending My Approval Logic

A user sees requests pending their action when their display name matches the NEXT_APPROVER_NAME field. The Dashboard's Approval Pipeline shows all in-flight requests to all users.

Hierarchy Management

Adding New Users

To add a new user to the approval chain, insert a row into USERS with:

FieldRequirement
SNOWFLAKE_USERNAMEMust match Snowflake login (uppercase, first initial + last name)
ROLEDM, RD, AVP, or GVP
APPROVAL_LEVEL2, 3, 4, or 5
MANAGER_NAMEDirect manager's display name
IS_FINAL_APPROVERtrue only for the GVP

The user's Workday data (title, manager, employee ID) is automatically resolved via VW_CURRENT_USER_INFO when they log in.

Current Hierarchy Gaps

The USERS table currently has only 2 entries (Level 1 AE and Level 5 GVP). Users at intermediate levels (DM, RD, AVP) are not yet configured. The approval chain still functions — any authenticated user can approve at any level — but NEXT_APPROVER_NAME routing and level enforcement depend on the USERS table configuration.
